# Интеграция Aeria Core с D&D 5e

Этот документ фиксирует главный технический принцип модуля: **Aeria Core не является отдельной игровой системой Foundry**. Модуль накладывает правила Аэрии поверх существующей системы `dnd5e` и должен работать как add-on module для нового чистого мира Foundry.

## Базовая архитектура

Aeria Core использует стандартную систему D&D 5e как основу.

```text
Foundry VTT
└── D&D 5e System
    └── Aeria Core Module
        ├── контент мира Аэрии
        ├── реакции и боевой импульс
        ├── точные попадания и слабые места
        ├── ритуалы и откаты
        ├── духи и обеты
        ├── фракционная репутация
        └── панель мастера
```

Модуль не должен заменять лист персонажа, базовые характеристики, HP, AC, навыки, спасброски, классы, предметы, заклинания, инициативу и стандартные механики бросков D&D 5e, если для этого нет отдельного утверждённого решения.

## Что остаётся за D&D 5e

Система `dnd5e` отвечает за базовую механику:

```text
характеристики;
навыки;
спасброски;
классы и подклассы;
расы / происхождения;
предметы;
оружие;
броню;
заклинания;
HP;
AC;
инициативу;
обычные атаки;
обычный урон;
обычные состояния;
активные эффекты;
лист персонажа.
```

Aeria Core не должен ломать эти сущности. Он должен добавлять дополнительные данные, кнопки, проверки, эффекты и подсказки поверх стандартной логики.

## Что добавляет Aeria Core

Aeria Core добавляет слой правил мира Аэрии:

```text
восстановление реакции через 1d10;
уклонение;
парирование;
контратака;
точные попадания;
слабые места существ;
проверки знания анатомии;
ритуалы для любого класса;
некромантические откаты;
духи и обеты;
фракционная репутация;
региональные события;
эпические действия;
панель мастера.
```

Эти механики должны быть отключаемыми через настройки модуля, где это возможно. Если мастер не хочет использовать конкретный блок правил, модуль не должен мешать вести игру стандартным способом.

## Принцип наложения механик

Модуль не отменяет стандартное действие D&D 5e, а добавляет к нему выбор или последствие.

Пример атаки:

```text
1. Игрок выбирает стандартную атаку на листе D&D 5e.
2. Aeria Core предлагает выбрать режим: обычная атака или точное попадание.
3. D&D 5e выполняет бросок атаки и урон.
4. Aeria Core применяет дополнительный эффект, если условия точного попадания выполнены.
```

Пример реакции:

```text
1. В начале хода персонажа модуль предлагает бросить 1d10.
2. На 1–4 реакция становится доступной.
3. На 5–10 реакция не восстанавливается.
4. Если реакция доступна, персонаж может потратить её на действие Аэрии.
5. После использования модуль отмечает реакцию как потраченную.
```

Пример ритуала:

```text
1. Персонаж открывает карточку ритуала.
2. Модуль показывает требования: инструкция, слова, компоненты, место, время, риск.
3. Мастер подтверждает, что условия выполнены или частично выполнены.
4. Выполняется проверка.
5. Модуль предлагает результат: успех, частичный успех, провал или откат.
```

## Хранение данных

Все дополнительные данные Aeria Core должны храниться в namespace модуля через flags Foundry.

Рекомендуемый namespace:

```text
aeriya
```

Примеры флагов:

```text
flags.aeriya.reaction.available
flags.aeriya.reaction.lastRoll
flags.aeriya.reaction.usedThisRound

flags.aeriya.targeting.knownWeakSpots
flags.aeriya.targeting.lastTargetedZone

flags.aeriya.rituals.knownRituals
flags.aeriya.rituals.activeBacklash

flags.aeriya.spirits.oaths
flags.aeriya.spirits.favor
flags.aeriya.spirits.debt

flags.aeriya.factions.reputation
flags.aeriya.factions.debts
```

Флаги должны быть читаемыми и предсказуемыми, чтобы мастер мог понять, что хранится в персонаже, NPC, монстре, предмете или сцене.

## Реакции и стандартная реакция D&D

В D&D 5e уже существует стандартная реакция. Механика Аэрии не должна бездумно конфликтовать с ней.

На этапе MVP реакция Аэрии считается отдельным ресурсом модуля:

```text
D&D Reaction — стандартная реакция системы dnd5e.
Aeria Reaction — очко реакции по правилам Аэрии.
```

В дальнейшем можно решить, объединять ли эти ресурсы. На старте безопаснее держать их отдельно, чтобы не ломать стандартные способности классов, заклинания и эффекты, которые уже используют реакцию D&D 5e.

## Точные попадания

Точные попадания не должны заменять базовую атаку.

Они работают как модификатор к обычной атаке:

```text
обычная атака D&D 5e
+ выбор зоны поражения
+ возможный штраф / сложность
+ дополнительный эффект при успехе
+ риск при провале
```

У каждого существа могут быть свои слабые места. Они должны храниться в карточке монстра или NPC и отображаться мастеру. Игрокам слабые места показываются только после наблюдения, проверки, изучения или сюжетного открытия.

## Ритуалы

Ритуалы не являются обычными заклинаниями D&D 5e.

Ритуал в Aeria Core — отдельная сущность мира. Его может провести любой персонаж при выполнении условий, даже если у него нет класса мага.

Ритуал не должен автоматически добавляться в список заклинаний персонажа. Он должен храниться как:

```text
JournalEntry;
Item custom type, если будет выбран безопасный способ расширения;
или карточка Aeria Core внутри compendium pack.
```

На этапе MVP безопаснее использовать JournalEntry/handout-формат и отдельную панель модуля, а не глубоко менять типы предметов D&D 5e.

## Некромант

Некромантия Аэрии должна разрабатываться как отдельный блок правил.

На старте некромант фиксируется как будущий класс или подкласс колдуна, но до утверждения правил не нужно вмешиваться в системные классы D&D 5e.

Безопасный подход:

```text
1. Сначала описать некроманта в docs/necromancer.md.
2. Затем собрать его способности как правила и handouts.
3. После тестов решить, делать ли subclass data для dnd5e.
4. Только после этого добавлять автоматизацию Foundry.
```

## Эпические действия

Эпические действия — это инструмент мастера, а не автоматическая кнопка для обхода правил.

Модуль может предлагать мастеру:

```text
тип действия;
пример сложности;
возможную цену;
последствие успеха;
последствие частичного успеха;
последствие провала.
```

Финальное решение всегда остаётся за мастером.

## Панель мастера

Панель мастера должна быть основным интерфейсом Aeria Core.

Она не должна заменять листы D&D 5e, но должна давать быстрый доступ к механикам Аэрии:

```text
бросить восстановление реакции;
открыть точное попадание;
посмотреть слабые места цели;
создать эпическое действие;
открыть ритуал;
бросить откат;
посмотреть духа;
проверить обеты;
открыть фракционную репутацию;
сгенерировать дорожное событие.
```

## Правило безопасности разработки

Любая механика Aeria Core должна проходить проверку:

```text
1. Она не ломает стандартный лист D&D 5e.
2. Она не перезаписывает системные данные без необходимости.
3. Она хранит свои данные в flags.aeriya.
4. Её можно отключить или игнорировать мастером.
5. Она помогает вести игру быстрее, а не замедляет сцену.
```

Если механика требует глубокого изменения системы D&D 5e, она выносится в отдельное обсуждение и не попадает в MVP без тестов.
