# MVP Aeria Core

Этот документ фиксирует границы первой рабочей версии модуля Aeria Core.

Главный принцип MVP: модуль должен быть полезен мастеру в новом мире Foundry на D&D 5e, но не должен сразу пытаться автоматизировать всю систему Аэрии.

## Цель MVP

Первая версия должна дать:

```text
рабочий manifest Foundry-модуля;
базовый JS API модуля;
настройки включения механик;
каркас панели мастера;
структуру контента;
шаблоны карточек;
правила реакций;
заготовку точных попаданий;
заготовку ритуалов;
заготовку эпических действий;
документацию для переноса контента;
первые чистые карточки мира.
```

MVP не должен быть финальным игровым продуктом. Он должен быть устойчивым фундаментом.

## Что автоматизируется в MVP

### Реакции

В MVP автоматизируется:

```text
бросок 1d10 на восстановление реакции;
порог восстановления реакции;
запись результата в flags.aeriya;
расход реакции;
выбор типа реакции;
сообщение в чат;
настройка включения / отключения механики.
```

Не автоматизируется в MVP:

```text
автоматическое снижение урона от уклонения;
автоматический расчёт парирования;
автоматическая контратака;
автоматическая привязка к конкретной атаке врага;
автоматический сброс в начале каждого раунда.
```

На первом этапе мастер сам решает, как именно уклонение, парирование или контратака повлияли на сцену.

## Точные попадания

В MVP автоматизируется:

```text
хранение слабых мест существа;
получение списка слабых мест;
фиксация выбранной зоны;
справочные эффекты по частям тела;
шаблон карточки монстра со слабыми местами.
```

Не автоматизируется в MVP:

```text
автоматическое изменение AC;
автоматическое добавление штрафа к атаке;
автоматическое наложение состояний;
автоматическое повреждение брони;
автоматическое падение, ослепление или обездвиживание.
```

Точные попадания в MVP являются инструментом мастера. Модуль помогает выбрать цель и описать возможный эффект, но не навязывает результат.

## Ритуалы

В MVP автоматизируется:

```text
список требований ритуала;
случайный пример отката;
запись последнего ритуала;
запись исхода ритуала;
добавление активного отката в flags.aeriya;
шаблон карточки ритуала.
```

Не автоматизируется в MVP:

```text
полная система компонентов;
расход компонентов из инвентаря;
автоматический расчёт сложности;
автоматическое вмешательство духов;
автоматическое создание долгов;
автоматическое изменение сцены.
```

Ритуалы в MVP работают как мастерский инструмент, а не как готовая spell-система.

## Некромантия

В MVP некромантия фиксируется как отдельный блок правил и будущий класс / подкласс колдуна.

Автоматизируется только:

```text
документация;
карточки ритуалов;
заготовки флагов;
учёт патрона;
учёт долга души;
учёт нарушения погребального порядка.
```

Не автоматизируется:

```text
создание полноценного класса;
создание subclass data для dnd5e;
новые заклинания;
автоматическое воскрешение;
автоматический обмен душ;
баланс способностей.
```

Некромант сначала проектируется в документации и тестируется вручную.

## Духи и обеты

В MVP закладываются:

```text
структура карточек духов;
флаги расположения духов;
флаги обетов;
флаги долгов;
шаблоны последствий.
```

Не автоматизируется:

```text
автоматическое наказание за нарушение обета;
автоматическое изменение репутации у духов;
автоматическая генерация знаков и примет;
полная система благословений.
```

## Фракции

В MVP закладываются:

```text
шаблон карточки фракции;
репутация через flags.aeriya;
диапазон отношения от -3 до +3;
структура долгов и доступов;
чистая папочная структура по Осколкам.
```

Не автоматизируется:

```text
автоматическое изменение репутации после каждого действия;
система фракционных войн;
экономические последствия;
динамическая карта влияния.
```

## Контент мира

В MVP не копируется старый мир `shadow-lands`.

Контент собирается заново из актуальной документации Аэрии.

Исключения из старого мира допускаются только точечно, если старая версия конкретной сущности точнее. Пример: город Окрам.

## Что не входит в MVP

В первую версию не входят:

```text
карты;
токены;
портреты;
аудио;
внешние компендиумы;
готовые визуальные сцены;
полная автоматизация боя;
полная автоматизация ритуалов;
самостоятельная system-разработка;
конвертация старого мира shadow-lands;
импорт старых LevelDB compendium packs как есть.
```

## Критерий готовности MVP

MVP считается готовым, когда:

```text
модуль устанавливается в новый мир Foundry;
модуль активируется без ошибок;
настройки отображаются в Foundry;
API game.aeriya доступен в консоли;
реакцию можно бросить и потратить;
слабые места можно записать и прочитать;
ритуальный откат можно сгенерировать;
структура контента понятна;
первые карточки мира перенесены в чистом формате.
```

## Главный риск

Главный риск — попытаться сделать сразу полноценную замену Foundry или D&D 5e.

Этого делать не нужно.

Aeria Core должен быть модулем-надстройкой. Сначала он помогает мастеру вести игру. Потом постепенно автоматизирует те части, которые реально повторяются и мешают скорости сессии.
